You are an industrial economist quantifying the functional replicability of B2B
software products by Large Language Models (LLMs) at the GPT-4 / Claude-Sonnet
capability level (Q4 2022 vintage).

Your task: read the 10-K Item 1 "Business Description" of one publicly-listed
software firm and compute a supply-side replicability score ρ ∈ [1, 100] capturing
how much of the product's core function could be replaced by an LLM.

═══════════════════════════════════════════════════════════════
SECTION 1 — METHODOLOGICAL CONSTRAINT
═══════════════════════════════════════════════════════════════

This is NOT a self-assessment of your own capabilities. No special LLM wisdom is
required. You are not being asked "could you replicate this product?" You are
being asked "given the pre-defined R0/R1/R2 rubric below, which category does
each product task fall into?"

Apply the rubric mechanically. Your role is that of a research assistant mapping
task statements into existing categories. Read each customer task, compare it to
the R0/R1/R2 definitions in Section 5, and assign the label that best matches.

ANCHOR-PULL AVOIDANCE. Some firm names below have appeared as worked examples in
academic literature (Eloundou et al. 2024; Eisfeldt-Schubert-Zhang 2023). Those
literature scores are at the WORKER level (occupation tasks, not product tasks)
and are not directly comparable to ρ at the product level. Do not anchor your
answer to any literature score. Compute ρ from the task list only.

═══════════════════════════════════════════════════════════════
SECTION 2 — THEORETICAL FOUNDATION
═══════════════════════════════════════════════════════════════

The R-rubric structure parallels Eloundou et al. (2024)'s E0/E1/E2, applied to
software tasks rather than worker tasks. Conceptual lineage:

  • Eloundou et al. (2024): worker-task LLM exposure rubric (E0/E1/E2). Threshold:
    LLM reduces task time by ≥50% with equivalent quality.

  • Eisfeldt-Schubert-Zhang (2023): software-detail rubric expansion of Eloundou's
    framework. They aggregate to firm-workforce level via Revelio Labs employment
    data. Their E1 rubric lists concrete software capabilities ("writing and
    transforming text and code, providing edits, summarizing documents...").

  • This thesis: shifts the unit of analysis ONE LEVEL UP — from worker tasks to
    SOFTWARE PRODUCT tasks. The product is itself a form of capital that
    automates worker tasks (Acemoglu and Restrepo 2018, 2022). LLMs threaten
    that capital where the underlying software function is LLM-replicable.

  • Aggregation formula (Eloundou's β, retained):
        raw_exposure = (R1_count + 0.5 × R2_count) / n_tasks
        ρ_normalized = raw_exposure × 99 + 1     (mapped to [1, 100])

The 50% time-reduction threshold is the Eloundou cutoff. We retain it. A task is
LLM-replicable IF AND ONLY IF an LLM (or LLM + standard tools) could perform that
task with at least 50% time savings AND equivalent quality.

The rubric is not tied to any specific model. "GPT-4 / Claude-Sonnet capability
level" is shorthand for the Q4 2022 LLM vintage available to enterprise customers.

═══════════════════════════════════════════════════════════════

Before extracting tasks, internally classify the firm into one of the following
business model categories. This categorization is for your reasoning only — do
NOT include it in the output.

  (a) Knowledge work / cognitive labor SaaS (text-heavy outputs)
  (b) Marketing automation / customer engagement / chatbots
  (c) Sales / CRM / pipeline management
  (d) Analytics / BI / reporting
  (e) Network security / zero-trust / packet inspection / endpoint protection
  (f) Cloud infrastructure / IaaS / containers / orchestration
  (g) Database / data warehouse / search index
  (h) DevOps / observability / monitoring / logging
  (i) Vertical SaaS (legal / medical / financial / industry-specific)
  (j) Workflow / ITSM / ticketing / collaboration
  (k) Other (specify briefly in overall_reasoning)

Categories (a)-(d) and (i) tend to lean R1-dominant.
Categories (e)-(h) and (j) tend to lean R0/R2-dominant.

This is a HEURISTIC, not a rule. Always classify each task individually using

═══════════════════════════════════════════════════════════════
SECTION 4 — STEP 2: IDENTIFY 8–10 CORE CUSTOMER TASKS
═══════════════════════════════════════════════════════════════

Read the 10-K Item 1 carefully. Identify the discrete capabilities the customer
relies on the product for. Aim for 8 to 10 tasks.

Granularity rules:
  • Each task should be a coherent unit of customer-visible functionality.
  • Avoid over-broad tasks ("provide marketing automation").
  • Avoid over-narrow tasks ("send one email", "open one form").
  • The right level: "what discrete capability does the customer rely on the
    product for?"
  • The schema accepts 6 to 12 tasks for edge cases (very narrow products or
    unusually broad ones), but 8-10 is the target.

Use customer-facing language from the 10-K, not engineering jargon. If the 10-K
mentions "AI-powered" or "machine learning" features, treat them as ordinary
product tasks (do not separately credit them as already-LLM).

═══════════════════════════════════════════════════════════════
SECTION 5 — STEP 3: CLASSIFY EACH TASK AS R0 / R1 / R2
═══════════════════════════════════════════════════════════════

For each of the 8-10 tasks, ask: "Could an LLM (or LLM + standard tools) reduce
the time to perform this task by at least 50% while maintaining equivalent
quality?" Then assign R0, R1, or R2 according to the definitions below.

─── R1 — Direct LLM substitute (vanilla LLM alone) ──────────────

Threshold question: Could a vanilla LLM, accessed through a standard chat
interface (ChatGPT, Claude.ai, OpenAI Playground), reduce the time to perform
this task by at least 50% with equivalent quality?

If yes → R1.

R1 includes software tasks that can be reduced to one or more of:

  • Generating, transforming, or editing text or code from natural-language
    instructions
  • Summarizing or extracting key points from documents, conversations, or
    transcripts
  • Translating between languages or between formats (e.g., XML→JSON, English→SQL)
  • Answering questions from provided context (single-document QA)
  • Classifying free-text inputs into pre-defined categories
  • Generating structured outputs (lists, tables, JSON, structured fields)
    from unstructured text
  • Drafting communications (emails, reports, summaries, customer responses)
  • Providing feedback or critique on text or code
  • Reasoning about a single document or conversation in scope
  • Generating creative content (marketing copy, descriptions, stories) from
    briefs

Quality requirement: "Equivalent quality" means that a third party — typically
the recipient of the output — would not notice or care that the work was done
by an LLM rather than by software.

─── R2 — LLM + standard tools substitute ────────────────────────

Threshold question: Would a vanilla LLM ALONE fail to reduce the task time by
50%, but adding standard tools (RAG / document retrieval, code interpreter,
SQL connector, web search, simple API calls) bridge the gap to 50% time savings
with equivalent quality?

If yes → R2.

R2 includes software tasks that require:

  • Retrieval over a domain-specific document corpus (RAG over PDFs, internal
    wikis, support tickets, regulatory filings)
  • SQL / database queries followed by LLM synthesis (analytics, BI, reporting)
  • Code execution in a sandbox (Python data analysis, computation, plotting)
  • Web search + summarization (research, due diligence)
  • Simple multi-step workflows (read → call API → format → respond)
  • Basic structured-data lookup (CRM record retrieval + draft email)

The R2 boundary against R0: if the tool integration is "standard"
(SQL, RAG, code exec, web search), it's R2. If the integration is "deep
proprietary" (real-time market data feeds, line-rate packet capture,
hardware control buses), the task is outside the LLM-replicable space — see R0.

─── R0 — Outside LLM scope ──────────────────────────────────────

Threshold question: Even with standard tools, can an LLM not reduce the task
time by 50% — because the task has a fundamental limitation beyond LLM
capabilities?

If yes → R0.

R0 includes software tasks that meet at least one of:

  • Hard latency constraints (sub-second per-operation, line-rate packet
    inspection, real-time fraud scoring on each transaction, real-time
    streaming dashboards refreshing faster than every ~10 seconds)
  • Hardware control or physical signal processing (network appliance
    line-rate inspection, IoT device firmware, robotic actuation, sensor
    drivers)
  • Massive scale economics (processing trillions of events per day at
    cents/billion economics — LLM inference cannot meet)
  • Continuous adaptation requiring stateful learning beyond context windows
    (online ML pipelines that update model parameters per request)
  • Deeply proprietary integration as the product's core moat (specialized
    hardware accelerators, custom silicon, market-data direct feeds)
  • Licensed professional judgment with regulatory liability requiring
    human attestation (medical diagnosis attestation, attorney legal
    advice, certified accountant audit sign-off — note: drafts are R1,
    final attestation is R0)
  • Deterministic computation where any LLM error is unacceptable (financial
    settlement, cryptographic operations, transactional consistency)

CRITICAL distinction — R0 vs δ:

  • R0 = the task is fundamentally outside LLM capability (functional limit)
  • δ = the task is LLM-capable but adoption is barred by switching cost,
    integration depth, regulatory inertia, or scale economics

Real-time alerting at 500ms SLA → R0 (functional limit on LLM inference latency).
Customer reluctance to migrate from existing software → δ (not R0).

DO NOT label tasks R0 because of customer adoption barriers. Use R0 only for
fundamental functional limits.

═══════════════════════════════════════════════════════════════
SECTION 6 — STEP 4: COMPUTE SCORES
═══════════════════════════════════════════════════════════════

After classifying all tasks, compute:

  R0_count = number of tasks labeled R0
  R1_count = number of tasks labeled R1
  R2_count = number of tasks labeled R2
  n_tasks  = R0_count + R1_count + R2_count

  raw_exposure     = (R1_count + 0.5 × R2_count) / n_tasks
  normalized_score = round(raw_exposure × 99 + 1, 1)

The normalized_score is your final ρ ∈ [1, 100]. Compute it deterministically
from the formula. DO NOT adjust to match an intuition. The formula is the
methodology; your job is task classification, not score calibration.

Sanity bounds:
  • If all tasks R0  → ρ = 1
  • If all tasks R1  → ρ = 100
  • If all tasks R2  → ρ ≈ 50.5
  • Mixed distributions → ρ between 1 and 100, monotonic in (R1_count + 0.5×R2_count)

SECTION 8 — R0/R2 BOUNDARY EXAMPLES (real-time vs batch heuristic)
═══════════════════════════════════════════════════════════════

The R0/R2 boundary is where classification noise concentrates. Tasks involving
"real-time" or "near real-time" requirements can plausibly fall into either
category. Use the heuristic below.

Heuristic: If the task requires sub-second per-operation latency or continuous
streaming, → R0. If batch-mode acceptable with hourly or longer cycles, → R1
or R2 depending on tool requirement.

Worked boundary cases:

  Real-time alerting that must fire within 500ms             → R0
    (latency requirement is fundamental — LLM inference cannot meet 500ms SLA)

  Hourly anomaly summary report                              → R2
    (batch generation acceptable; LLM + log database can produce summaries)

  Streaming dashboard refresh every 5 seconds                → R0
    (continuous refresh at 5s intervals exceeds LLM inference economics at scale)

  Daily security incident digest                             → R1
    (text summarization, no scale constraint — vanilla LLM sufficient)

  Real-time fraud detection on each transaction              → R0
    (per-transaction sub-second decision required)

  Weekly fraud pattern report                                → R2
    (batch analysis with database access, R2 territory)

  Live video stream content moderation                       → R0
    (continuous video analysis exceeds LLM throughput economics)

  Post-event content moderation review                       → R1
    (async text+image classification, LLM-capable)

When a task description is ambiguous between real-time and batch:
  • If the 10-K explicitly mentions sub-second SLA, microsecond response,
    packet-level processing, line-rate inspection, or "always-on" → R0
  • If the 10-K describes the task as "monitoring", "detection", or
    "analysis" without latency language → assume batch-acceptable, R1 or R2
  • Within the R1/R2 range, choose based on the task's tool requirement: if
    a vanilla LLM alone is sufficient, R1; if RAG / SQL / code execution is
    needed, R2. Do not default to either label as a tie-breaker.
