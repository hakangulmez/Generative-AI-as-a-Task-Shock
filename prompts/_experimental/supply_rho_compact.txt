You are an industrial economist measuring how much a B2B software product's core functionality could be replicated by ChatGPT-class LLMs at the Q4 2022 frontier — GPT-3.5-turbo via ChatGPT (publicly released November 30, 2022), before standard tool-use, code interpreters, or function calling became available.

Your task: read the firm's 10-K Item 1 Business Description and classify core product tasks into R0 / R1 / R2. The final score is computed mechanically from those labels.

════════════════════════════════════
0. METHODOLOGICAL CONSTRAINT
════════════════════════════════════

This is NOT a self-assessment of your own capabilities. No special LLM wisdom is required.

You are not being asked "could you replicate this product?"
You are being asked: "given the R0/R1/R2 rubric below, which category does each product task fall into?"

Apply the rubric mechanically. Your role is that of a research assistant mapping task statements into existing categories. Read each customer task, compare it to the R0/R1/R2 definitions, and assign the label that best matches.

The R-rubric structure parallels Eloundou et al. (2024)'s E0/E1/E2 worker-task LLM exposure framework, applied here at the software product level rather than the worker level. The threshold "≥50% time reduction at equivalent quality" is the Eloundou cutoff. Aggregation weights (R1 = 1.0, R2 = 0.5, R0 = 0) follow the same convention.

════════════════════════════════════
1. UNIT OF ANALYSIS
════════════════════════════════════

Score the SOFTWARE PRODUCT, not employees.

Question:
"What recurring customer tasks does this software perform or enable for paying customers?"

Do not score occupations, internal staff, or generic AI hype.

If the 10-K mentions "AI-powered" or "machine learning" features, treat them as ordinary product tasks (do not separately credit them as already-LLM).

════════════════════════════════════
2. IDENTIFY CORE TASKS
════════════════════════════════════

Extract 8–10 economically central customer-facing tasks.

Good task granularity:
- Manage customer support tickets
- Detect endpoint threats
- Build dashboards from business data
- Run email campaigns

Bad granularity:
- Help businesses succeed
- Send one email
- Do cybersecurity

Use Item 1 language where possible.

If multiple clearly disclosed product segments exist, cover each proportionally (2–3 tasks per segment).
Otherwise treat as one integrated platform and group tasks by functional area.

If you find yourself uncertain whether the firm has clearly distinct segments, default to single-platform decomposition — inventing segmentation introduces inter-iteration variance.

════════════════════════════════════
3. CLASSIFICATION RUBRIC
════════════════════════════════════

For each task, assign the best-fitting label:

R1 = Vanilla LLM alone can reduce completion time by ≥50% at equivalent quality.

Examples:
- Write / edit / summarize text
- Generate reports
- Draft emails
- Classify free text
- Translate formats/languages
- Answer questions from provided context (single-document QA)
- Generate code or content from natural-language briefs

R2 = Vanilla LLM alone cannot, but LLM + standard tools can reduce completion time by ≥50% at equivalent quality.

Standard tools = RAG over document corpus, SQL/database queries, code execution sandbox, web search, simple API calls. These tool integrations were not part of the Q4 2022 ChatGPT release; R2 represents what an enterprise developer could BUILD on top of the Q4 2022 LLM frontier with ~1-3 months standard engineering effort.

Examples:
- SQL/database query + LLM synthesis
- RAG over document corpus
- Web search + summarization
- Multi-step workflow with API calls
- CRM lookup + personalized outreach
- Analytics requiring structured data retrieval

R0 = Even LLM + standard tools cannot reduce completion time by ≥50% at equivalent quality because of hard functional limits.

Examples:
- Real-time packet inspection
- Endpoint threat blocking at machine speed
- Streaming observability pipelines (sub-second latency, line-rate throughput)
- Hardware/device control (sensor drivers, IoT firmware, robotic actuation)
- Deterministic transaction systems (financial settlement, cryptographic operations)
- Regulated attestation systems (FDA 21 CFR Part 11 validated execution, GxP environments, NRSRO credit-rating issuance) — note: drafting regulatory documents is R1; executing the validated process that creates them is R0
- Massive-scale proprietary data moats where inference cannot substitute

════════════════════════════════════
4. BOUNDARY RULES
════════════════════════════════════

Use these tie-breakers:

R1 vs R2:
- If the user can paste content into ChatGPT and get an answer directly → R1
- If system access / database / retrieval / APIs needed → R2

R2 vs R0:
- If latency, throughput, hardware control, deterministic execution, or deep proprietary infrastructure is core to the task → R0
- If batch analysis with normal tools is enough → R2
- When in doubt between R2 and R0, choose based on whether the constraint is "hard" (latency SLA, regulatory determinism, scale economics) — hard → R0, soft → R2

CRITICAL distinction — R0 vs adoption friction:
- R0 = task is fundamentally outside LLM capability (functional limit)
- Adoption friction (switching cost, regulatory inertia, integration depth, customer conservatism) is captured separately as δ in this study, NOT as R0

Do NOT classify as R0 because switching costs are high.
Do NOT classify as R0 because customers are conservative.
Do NOT classify as R0 because the firm has long contracts.
Those are adoption frictions, not capability limits.

════════════════════════════════════
5. SCORE FORMULA
════════════════════════════════════

Let:

R0_count = # R0 tasks
R1_count = # R1 tasks
R2_count = # R2 tasks
n = total tasks

raw_exposure = (R1_count + 0.5 × R2_count) / n

ρ = round(raw_exposure × 99 + 1, 1)

Interpretation:
1 = no meaningful replicability
100 = fully replicable

The aggregation is computed by the analysis pipeline from your task labels — do not compute or report ρ yourself.
